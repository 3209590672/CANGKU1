# 编程准则

> 基于 OdysseyDemo 架构实践提炼，适用于 Unity 3D 游戏开发。

---

## 一、四层架构

每个游戏项目按此四层组织代码：

```
Layer 4 — 内容/资产层         Models, Animations, Prefabs, Scenes
Layer 3 — 游戏特定逻辑        Player/Enemy 具体实现、关卡控制
Layer 2 — 可复用模式          输入封装、音频事件、动画同步、镜头
Layer 1 — 基础框架            泛型 FSM、Entity 控制器、Stats、事件总线
```

**规则**：Layer 1 零游戏依赖，可跨项目移植。Layer 3 每款游戏重写。Layer 2 设计模式复用，实现按需适配。

---

## 二、七大设计原则

### 1. 单一职责原则（SRP）

> 一个类只负责一件事。

**做法**：

```
Player.cs 不自己处理输入/动画/音效/特效/镜头，
而是组合六个独立组件：

Player
  ├── PlayerInputManager    ← 只负责输入
  ├── PlayerAnimator        ← 只负责动画
  ├── PlayerAudio           ← 只负责音效
  ├── PlayerParticles       ← 只负责特效
  ├── PlayerCamera          ← 只负责镜头
  └── PlayerStatsManager    ← 只负责属性数据
```

状态类同样原子化：`IdleState`、`WalkState`、`DashState` 各管各的，互不干涉。

**检测标准**：能用一句话说清楚这个类是干什么的。说不清就拆。

---

### 2. 开闭原则（OCP）

> 扩展新功能不修改已有代码。

**做法 — 新增状态时不改状态机**：

```csharp
// EntityStateManager 定义一次，永不改动
public virtual void Change<TState>() where TState : EntityState<T>
{
    if (m_states.ContainsKey(typeof(TState)))
        Change(m_states[typeof(TState)]);
}

// 新增状态只需新建类
public class FlyPlayerState : PlayerState { ... }

// 调用方
player.states.Change<FlyPlayerState>();  // ← 框架代码一行不改
```

**做法 — 新增交互物不改物理系统**：

```csharp
public interface IEntityContact
{
    void OnEntityContact(EntityBase entity);
}

// Entity<T> 的碰撞检测只依赖接口
var contact = collider.GetComponent<IEntityContact>();
contact?.OnEntityContact(this);  // ← 这行永远不变

// 新增交互物：实现接口即可
public class Trampoline : MonoBehaviour, IEntityContact { ... }
public class ConveyorBelt : MonoBehaviour, IEntityContact { ... }
```

**做法 — 新增难度不改逻辑代码**：

```csharp
// 属性抽到 ScriptableObject，拖拽 .asset 文件换配置
player.stats.Change(1);  // 切到 HardMode.asset，代码不动
```

### 3. 里氏替换原则（LSP）

> 子类能无差别替换父类。

**做法**：

```csharp
// 高层代码操作抽象类型
public class EntityHitbox : MonoBehaviour
{
    private EntityBase m_entity;

    public void Attack()
    {
        m_entity.ApplyDamage(damage, origin);
        // ↑ 传入 Player 或 Enemy 都正确 — LSP 保证
    }
}

// 子类重写方法保持签名和语义一致
public class Enemy : Entity<Enemy>
{
    public override void ApplyDamage(int damage, Vector3 origin)
    {
        health.Damage(damage);              // 不改变基类含义
        states.Change<HurtEnemyState>();    // 只扩展不扭曲
    }
}
```

**检测标准**：所有用到 `EntityBase` 的地方换成 `Player` 或 `Enemy` 都不会出错。

---

### 4. 依赖倒置原则（DIP）

> 依赖抽象接口，不依赖具体实现。

**做法**：

```csharp
// ✅ 正确 — 高层依赖接口
public abstract class Entity<T> : EntityBase
{
    void HandleContacts()
    {
        var contact = other.GetComponent<IEntityContact>();
        contact?.OnEntityContact(this);  // 依赖抽象
    }
}

// ❌ 错误 — 高层依赖具体类
void HandleContacts()
{
    var spring = other.GetComponent<Spring>();
    if (spring != null) spring.Bounce();
    var hazard = other.GetComponent<Hazard>();
    if (hazard != null) hazard.Hurt();
    // 每加一种交互物就要加一段 if — 没完没了
}
```

**规则**：模块间通信只通过 interface。`GetComponent<具体类>()` 只允许在创建对象的地方用一次。

---

### 5. 接口隔离原则（ISP）

> 接口要小而精，不要逼实现者写空方法。

**做法**：

```csharp
// ✅ 极小接口 — 只有一个方法
public interface IEntityContact
{
    void OnEntityContact(EntityBase entity);
}

// ❌ 不要搞大而全
public interface IEntityContact
{
    void OnEnter();   // ← Spring 不需要
    void OnExit();    // ← Hazard 不需要
    void OnStay();    // ← Pickable 不需要
}
```

**规则**：一个 interface 超过 3 个方法就要审视是否能拆。宁可多几个小接口，不要一个大接口。

---

### 6. 迪米特法则（LoD）

> 只和直接朋友说话，不碰陌生人。

**做法**：

```csharp
// ✅ 状态只和 player 通信
public class IdlePlayerState : PlayerState
{
    protected override void OnStep(Player player)
    {
        if (player.inputs.GetJumpDown())
            player.states.Change<FallPlayerState>();  // 请求切换，不关心切到哪
    }
}

// ❌ 越界访问
// player.GetComponent<Enemy>().Damage(10);      ← 不碰别的实体
// player.states.m_list[3].timeSinceEntered;      ← 不碰内层数据
// WalkPlayerState.Instance.SomeMethod();          ← 不碰其他状态
```

组件间通过事件通信，不直接引用：

```csharp
// Player 只触发自己持有的事件
playerEvents.OnJump.Invoke();

// 音频、特效各自订阅，Player 不知道它们的存在
playerEvents.OnJump.AddListener(PlaySound);
playerEvents.OnJump.AddListener(EmitParticles);
```

**规则**：`a.b.c.d` 链式调用不允许超过两节。超出就用方法封装或事件解耦。

### 7. 合成复用原则（CARP）

> 用组合（Has-A）而不是继承（Is-A）来复用代码。

**这是最重要的原则。**

**做法 — 组件组合而非继承链**：

```csharp
// ✅ 组合：Player 包含多个独立组件
Player (GameObject)
  ├── PlayerInputManager   (Has-A)
  ├── PlayerAnimator       (Has-A)
  ├── PlayerAudio          (Has-A)
  ├── PlayerParticles      (Has-A)
  ├── Health               (Has-A)
  └── PlayerStatsManager   (Has-A)
```

```csharp
// ❌ 继承：每加一个能力就要新建子类，组合爆炸
BasePlayer → JumpingPlayer → AnimatedJumpingPlayer → AudioAnimatedJumpingPlayer → ...
```

**做法 — 属性数据用 ScriptableObject 组合**：

```csharp
// ✅ 属性和实体是组合关系
public class Player : Entity<Player>
{
    public PlayerStatsManager stats;  // 属性是独立资产，可拖拽替换
}

// ❌ 属性写死在类里
public class Player : MonoBehaviour
{
    public float walkSpeed = 5f;   // 换数值 = 改代码
}
```

**做法 — 状态平级列表而非继承链**：

```csharp
// ✅ 状态是平级组合，增减只改数组
public string[] m_states = { "IdleState", "WalkState", "DashState" };
```

**规则**：优先问"我能把这个能力抽成独立组件吗？"能抽就抽。继承只用于"是"的关系（Player 是 Entity），不用于"有"的关系（Player 有健康值）。

## 三、检查清单

每次写新功能前过一遍：

- [ ] 这个类一句话能说清吗？（SRP）
- [ ] 加这个功能需要改已有代码吗？（OCP）
- [ ] 用子类替换父类会出错吗？（LSP）
- [ ] 依赖的是 interface 还是具体类？（DIP）
- [ ] 这个 interface 能再拆小一点吗？（ISP）
- [ ] 有跨两层的链式调用吗？（LoD）
- [ ] 能用组合代替继承吗？（CARP）
- [ ] 属性数据能抽成 ScriptableObject 吗？（CARP）

---

> **一句话总结**：小接口、多组合、事件解耦、数据外置、扩展不改源码。
