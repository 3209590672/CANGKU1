# Space Voyager — 代码架构

## 项目概况

```
游戏类型：3D 太空跑酷射击
总代码量：~7500 行 Lua
文件数量：34 个模块
架构模式：四层分层 + 薄调度入口
设计原则：SOLID（详见 docs/编程准则.md）
```

```
    655 行                5217 行                   625 行             218 行
```

**层级规则**：
- Layer 4：纯数据/配置，零逻辑代码，策划可直接调数字
- Layer 3：游戏特定逻辑，每款游戏重写
- Layer 2：设计模式复用，实现按需适配
- Layer 1：零游戏依赖，可跨项目直接移植

**依赖方向**（严格单向，禁止反向）：

```
Layer 4 (数据)     ← 被所有层读取，自身无依赖
    ▲
Layer 3 (游戏)     ← 读 Layer4 配置，调 Layer2 工具
    ▲
Layer 2 (模式)     ← 调 Layer1 基础设施
    ▲
Layer 1 (框架)     ← 零外部依赖
```

## 数据流模型

### 核心数据中心：gs 表

整个项目有一个中心化的可变状态表 `gs`，由 `GameState.new()` 创建，存活于 main.lua：

```lua
gs = {
    phase, gameTime, frameCount,        -- 流程
    score, lives, invincibleTimer,      -- 战斗
    speed, shipX, shipY, shipVelX/Y,    -- 运动
    shieldActive, shieldTimer, ...      -- 护盾
    warpEnergy, warpActive, ...         -- 折跃
    asteroids = {},                     -- 小行星实体列表
}
```

### 三种数据传递模式

```
┌──────────────────────────────────────────────────────────────┐
│                       main.lua 调度层                         │
│                                                              │
│   ① init(deps) — 启动时一次性注入不变依赖                      │
│      scene, 材质, 模型引用, 配置常量                            │
│                                                              │
│   ② update(dt, gs) — 每帧传递当前状态                          │
│      模块直接读写 gs 字段（就地修改，无返回值）                   │
│                                                              │
│   ③ 闭包回调 — 延迟绑定 + 反向通知                             │
│      getSpeed = function() return gs.speed end               │
│      onGameOver = function() GameOver() end                  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 通信规则

```
• 模块之间不直接通信（无横向 require）
• 所有协调通过 main.lua 的调用顺序保证
• 数据共享通过 gs 表（中心化可变状态）
• 反向通知通过闭包回调 (onGameOver, showUI...)
```

### 一帧的执行流程

```
HandleUpdate(dt)
│
├── 所有阶段：雾动画 / 星尘 / 装饰陨石 / 流星
│
├── STATE_MENU:
│   └── 飞船浮动 → EngineTrails.updateMenu → 按钮呼吸动画
│
├── STATE_PLAYING:
│   ├── UpdateShip        ← 飞船移动/翻滚/倾斜
│   ├── UpdateWarp        ← 折跃充能/激活判断
│   ├── UpdateShooting    ← 射击输入/热量
│   ├── Shield.update     ← 护盾计时/动画
│   ├── UpdateBullets     ← 子弹移动/回收
│   ├── UpdateAsteroids   ← 小行星生成/移动
│   ├── UpdateCrystals    ← 水晶生成/移动
│   ├── WarpVisuals.update← 折跃特效
│   ├── CheckCollisions   ← 碰撞→扣血/加分/触发死亡
│   ├── UpdateDifficulty  ← 难度递增
│   └── HUD.update        ← 更新显示 (每3帧)
│
└── STATE_GAMEOVER / STORY:
    └── 按钮呼吸动画
```

## 数据所有权

| 数据 | 存放位置 | 谁读 | 谁写 |
|------|---------|------|------|
| 游戏状态 (gs) | main local | 几乎所有模块 | main + Shield + Collisions + GameFlow |
| 子弹列表 | Weapons 内部 | Collisions (getter) | Weapons |
| 小行星列表 | gs.asteroids | Collisions, Asteroids | Asteroids |
| 水晶列表 | Crystals 内部 | Collisions (getter) | Crystals |
| 飞船节点 | main local | 多模块(参数传入) | main (position/rotation) |
| 拖尾数据 | EngineTrails 内部 | WarpVisuals (getter) | EngineTrails |
| 材质实例 | main locals | 各模块(init注入) | 只初始化时写一次 |
| UI 控件 | HUD 内部 | HUD | ShowPlayingUI 注入 |

## 设计原则对照

| 原则 | 实践方式 |
|------|---------|
| SRP 单一职责 | 每个文件一句话说清职责 |
| OCP 开闭 | 数值全在 config.lua，改配置不改逻辑 |
| DIP 依赖倒置 | 模块通过 init(deps) 注入，不 require 上层 |
| CARP 合成复用 | 纯组合架构，无继承链 |
| LoD 迪米特 | 模块间零直接引用，均经 main 转发 |

## 技术亮点

| 特性 | 实现方式 |
|------|---------|
| 飞船模型 | 纯代码程序化生成（无外部模型文件） |
| 小行星 | 6 种随机网格变体，预生成池化复用 |
| 引擎拖尾 | BillboardSet 光带 + 火花粒子系统 |
| 护盾 | CustomGeometry 动态球体 + 脉冲动画 |
| 折跃特效 | 32 条光条 + 翼尖拖尾 + 星空加速 |
| HUD | 脏标记缓存，仅值变化时更新 UI |
| 剧情系统 | 行为计数 → 分支触发 → 多结局 |
